    📘 DocStill 사용 상세 가이드 (운영자: 문성혁)¶

    quantinue-wiki/ 안에서의 모든 작업 방법. 복붙용 명령·지시 문구 포함. 구축 배경·시스템 동작 원리는 시스템 설명서 참조.

    0. 폴더 지도¶

    quantinue-wiki/
├ raw/                       원본 (불변 · 사람별 + 공용 + 멘토) ← 유일하게 "넣는" 곳
├ documents/                 사이트에 노출되는 것 (SSOT + 문서)
│  ├ facts/                  SSOT (증류·병입으로만 갱신 — 손으로 편집 금지)
│  ├ STATUS.md               현황판 (증류가 덮어씀)
│  ├ 질문.md                 회의 안건 (증류가 추가, 병입이 정리)
│  ├ 추천제안.md             AI 추천 제안 (증류가 매번 재생성)
│  ├ 산출물_현황판.md         제출물 11종 완성도
│  └ 사용가이드.md·시스템설명서.md   문서
├ archive/                   사이트에 안 보이는 파일
│  └ briefings/              (선택) 증류 보고 보관
├ tools/distill_prompt.md    증류 지시서 = 이 시스템의 소스코드
└ CLAUDE.md · .claude/       AI 하네스 — 세션 규칙 자동 로드 + /증류·/병입·/점검·/배포 명령

    대원칙: 내 손으로 직접 편집하는 파일은 없다. 나는 ①raw에 파일 넣기 ②클로드에게 지시 ③결과 검토, 세 가지만 한다. (facts를 손으로 고치고 싶어져도 → 병입 지시로 하기. 이력이 남아야 SSOT.)

    1. 파일이 들어왔을 때 (수시 · 10초)¶

    1. 공유방에서 다운로드
    2. raw/작성자이름/ 폴더에 넣기 — 팀 공용 문서는 raw/공용/, 멘토 피드백은 raw/멘토/
    3. 파일명은 자유. 정리하고 싶으면 YYYY-MM-DD_제목.html 권장 (필수 아님)

    같은 파일을 실수로 두 번 넣어도 OK — 증류가 내용 같으면 무시함.

    2. 증류 실행 (회의 전날 · ~20분)¶

    방법 A — 더블클릭: Finder에서 증류실행.command 더블클릭. 끝.

    방법 B — 터미널(CLI): 

    cd ~/Documents/ClaudeCode/quantinue-wiki
python3 tools/distill.py                    # 기본 백엔드(config.json)로 실행
python3 tools/distill.py --dry-run          # 새 파일 목록만 확인 (LLM 호출 X)
python3 tools/distill.py --backend ollama   # 백엔드 즉석 지정

    방법 C — 클로드 코드 대화로 (추천): claude 열고

    /증류 (슬래시 명령이 지시서 순서를 강제함. 결과를 보면서 조율 가능 — 추가 지시는 /증류 ○○만 주의해서 식으로)

    ⬇️ 어떤 LLM을 쓸지는 §2-B에서 상세히. 기본값(claude)로 그냥 돌려도 되고, 다른 LLM으로 바꾸려면 config.json 한 줄만 고치면 된다.

    📌 증류가 갱신하는 것: STATUS(현황판)·질문(안건)·추천제안(AI 제안, 매번 재생성)·업데이트로그. facts는 병입으로만.

    받은 보고에서 확인할 것 3가지: - ⚠️ 충돌이 진짜 충돌인가 (표현만 다른 동일 내용이면 "그건 동일 처리해줘") - 안건 문구가 회의에서 바로 쓸 수 있게 명확한가 - 주장 굵기가 이상하면 → 지시서에 규칙 추가 지시: "distill_prompt.md 굵기 기준에 '○○ 같은 건 제외' 추가해줘"

    끝나면 자동 커밋됨(config auto_git_commit). 수동이면: 

    git add -A && git commit -m "증류 N회차"

    2-B. LLM 백엔드 — "어떤 LLM을 쓰든" 운영법 (상세)¶

    왜 벤더 중립인가 (원리)¶

    이 시스템은 특정 LLM에 묶이지 않는다. 비결은 역할 분리다:

    • "무엇을 추출하나"(주장·4분류·스타일 규칙) = tools/distill_prompt.md 한 파일에만 있음 → 어떤 LLM에도 그대로 통함
    • "파일을 어떻게 읽고 쓰나" = tools/distill.py가 백엔드별로 감쌈

    그래서 LLM을 바꿔도 증류 품질 규칙은 그대로고, config.json의 "backend" 값만 바꾸면 된다.

    백엔드는 딱 두 종류(type)¶

    type=agent  : CLI 에이전트가 스스로 raw를 읽고 STATUS/질문.md를 씀 → 우리는 호출만
             └ claude, codex  (또는 프롬프트를 인자로 받는 아무 CLI 툴)
type=api    : LLM은 텍스트만. raw 읽기·마커 파싱·파일 쓰기는 전부 distill.py(파이썬)가 대행
             ├ api_ollama : ollama 네이티브(/api/generate) — 로컬 모델
             └ api_openai : OpenAI 호환(/v1/chat/completions) — 클라우드·로컬 대부분

    지원 백엔드 한눈에¶

    config 키 type 대상 비용/키 품질 언제
    claude (기본) agent 클로드 코드 구독 ★★★ 평소 · 품질 기준 잡기
    codex agent Codex CLI OpenAI 계정 ★★★ 코덱스 쓰는 경우
    ollama api_ollama 로컬(qwen 등) 무료·오프라인 ★~★★ 반복 실험 · 비용 0 · 프라이버시
    openai_compat api_openai OpenAI·Groq·Together·DeepSeek·로컬 vLLM·LM Studio·llama.cpp … API 키(로컬은 불필요) 서버따라 위 밖의 거의 모든 LLM

    openai_compat 하나가 OpenAI 호환 API를 말하는 모든 서버를 커버한다(클라우드·로컬 공통). 그래서 사실상 "어떤 LLM이든" 붙는다.

    백엔드 바꾸는 법 (3가지)¶

    ① 그때만 바꾸기 (즉석): 

    python3 tools/distill.py --backend ollama
python3 tools/distill.py --backend openai_compat

    ② 기본값 바꾸기: tools/config.json 맨 위 한 줄 

    { "backend": "ollama", ... }

    ③ 새 LLM/서버 추가: openai_compat의 url·model·api_key_env만 교체 

    "openai_compat": {
  "type": "api_openai",
  "url": "https://api.groq.com/openai/v1/chat/completions",   // ← 서버 주소
  "model": "llama-3.3-70b",                                    // ← 모델 이름
  "api_key_env": "GROQ_API_KEY"                                // ← 이 환경변수에서 키를 읽음
}
    (config.json 안 _예시_다른_서버에 Groq·Together·DeepSeek·vLLM·LM Studio 주소가 이미 적혀 있음)

    로컬 모델(ollama) 준비 — 무료·오프라인¶

    # 1) ollama 설치: https://ollama.com  →  2) 모델 받기:
ollama pull qwen3.6:35b-a3b-coding-mxfp8      # config의 기본 모델
# 3) 그냥 실행 (ollama가 백그라운드로 떠 있으면 됨)
python3 tools/distill.py --backend ollama

    API 키 넣는 법 (openai_compat, 클라우드일 때만)¶

    키를 코드·config에 절대 적지 않는다. 환경변수로 준다: 

    export OPENAI_API_KEY="sk-..."      # 터미널 세션에 (또는 ~/.zshrc 에 넣어두면 영구)
python3 tools/distill.py --backend openai_compat
    로컬 서버(vLLM·LM Studio)는 키가 필요 없어 api_key_env를 비우면 됨.

    품질·안전 팁¶

    • 초반은 claude(또는 강한 API)로 품질 기준을 잡고, 안정되면 ollama로 비용 0 반복.
    • api형은 LLM이 형식을 어기면(마커 깨짐) 파일을 안 건드리고 원문을 archive/briefings/에 저장 → 망칠 위험 0. 이럴 땐 강한 백엔드로 다시 돌리면 됨.
    • 어떤 백엔드로 돌리든 처리한 파일은 해시로 기억(tools/.state.json)하므로, 백엔드를 갈아타도 같은 파일을 두 번 처리하지 않는다.
    • 로컬 모델은 큰 HTML에서 컨텍스트가 부족할 수 있음 → config의 max_chars_per_doc로 문서당 글자 수를 제한(기본 60,000자, 초과분은 자동 잘림).

    보고에서 확인할 것 3가지 (백엔드 공통)¶

    • ⚠️ 충돌이 진짜 충돌인가 (표현만 다른 동일 내용이면 "그건 동일 처리해줘")
    • 안건 문구가 회의에서 바로 쓸 수 있게 명확한가
    • 주장 굵기가 이상하면 → 지시서에 규칙 추가: "distill_prompt.md 굵기 기준에 '○○ 같은 건 제외' 추가해줘"

    3. 회의에서 (기존 회의 시간)¶

    • 질문.md (또는 위키 안건 페이지) 화면공유 → 위에서부터 결정
    • 결정 안 나면 "보류"도 정상 — 억지로 확정하지 않기
    • 녹음하기 — 끝나면 전사(txt)를 raw/회의/YYYY-MM-DD_회의.txt로 저장 → 다음 /증류가 규격 회의록(결정·보류·액션아이템)을 만들고 "병입 대기 N건" 을 제시함 → 검토 후 "병입해줘" 한 마디면 결정로그·facts 반영 (§4와 합쳐짐)

    4. 회의 후 병입 (~10분)¶

    클로드 코드에게 (예시):

    /병입 안건 1번은 "MVP 성공 기준 = 한 사이클 자율 완주 + 결정저널 100% 근거"로 확정, 2번은 기각, 3번은 보류

    (/병입 명령이 결정로그 번호 → facts 반영 → 질문 제거 → 검증 순서를 강제함)

    → 검토 → git add -A && git commit -m "병입: 안건 1 확정"

    5. 주 1회 (10분)¶

    산출물_현황판.md 갱신해줘 — facts와 질문.md 기준으로 11종 완성도 재평가하고, 이번 주 채워야 할 🔴를 추천해줘

    5-B. 에이전트 코드 페이지 갱신 (에이전트 개발되면)¶

    에이전트 실제 코드는 별도 repo(~/Documents/ClaudeCode/quantinue/quantinue, 에이전트별 브랜치)에 있다. 위키의 핵심 기능 섹션(에이전트 페이지들)은 그 코드를 읽기 전용 참조해서 만든다 (그 repo엔 커밋 안 함).

    /에이전트갱신 critic ← 한 마디면 됨 (이름: reviewer·critic·strategist·info·core, 생략 시 갱신된 것 전부)

    • 브랜치를 읽기전용(git show)으로 읽고 → 페이지 갱신 + 참조: 브랜치@커밋 자동 추적 + 업데이트 로그 1줄.
    • 참조 커밋이 그대로면 "변경 없음"으로 끝 — 부담 없이 자주 돌려도 됨.
    • 함수 시그니처·라인 단위는 넣지 않는다(코드 바뀌면 낡음). 위키=이해, repo=진실.

    6. 산출물 조립 (제출 시점)¶

    facts/파이프라인.md, facts/데이터계약.md 를 재료로 [요구사항 정의서] 초안을 만들어줘. 모든 항목에 출처 남기고, 재료 없는 칸은 [미정]으로 비워둬. 지어내기 금지.

    • 초안 검토·문장 다듬기·강조점 선택 = 내 일 (여기가 사람의 가치)
    • 제출 후 멘토 피드백 오면 → raw/멘토/에 넣기 (다음 증류에서 자동 안건화)

    주차별 활동보고서 (거의 자동):

    git log와 facts/결정로그.md, STATUS.md 변화를 재료로 이번 주 활동보고서 초안 만들어줘 (진행 내용·R&R·이슈 형식)

    7. 위키 출판 (회의 후 1분)¶

    배포 = main에 push. GitHub Actions(.github/workflows/deploy.yml)가 lint → strict build → 배포를 자동 수행 — 검증 실패면 배포 자체가 안 되니 깨진 사이트가 올라갈 수 없다.

    mkdocs serve          # 로컬 미리보기 (회의 화면공유용) http://localhost:8000
git push origin main  # → 1~2분 뒤 라이브 반영
    클로드 코드로는 /배포 한 마디 — 로컬 lint·빌드 선검증 → 커밋 확인 → push → Actions 성공·라이브 200까지 확인함. (최초 1회 준비물은 pip install -r requirements.txt — 이미 구축 완료)

    8. 문제가 생기면¶

    증상 처방
    증류가 헛소리(없는 내용) 뽑음 출처를 물어봐라 — 출처 못 대면 삭제 지시. 지시서 "금지"에 사례 추가
    안건이 계속 중복됨 "질문.md에서 유사 안건 병합해줘" + 지시서에 병합 규칙 강화
    HTML이 커서 분석이 느림/부정확 "파이썬으로 텍스트 추출부터 해줘"
    facts를 실수로 손으로 고침 git diff로 확인 → 되돌리고 병입 절차로 다시
    한글 파일명이 검색에 안 걸림 macOS 유니코드 문제 — "폴더 나열해서 찾아줘"로 우회
    사이트가 어딘가 이상함 /점검 — lint.py(기계) + LLM(판단) 2단 감사. 급하면 python3 tools/lint.py만 몇 초
    배포가 안 올라감 GitHub Actions 탭에서 deploy-wiki 실패 로그 확인. Queued 멈춤이면 수동 재실행(Run workflow)
    lint가 정당한 걸 지적함(오탐) tools/lint.py의 CONFIG allow 목록에 추가 — 검사를 끄지 말 것
    뭔가 다 꼬인 것 같음 걱정 없음: raw는 불변이라 facts·STATUS 지우고 처음부터 재증류 가능

    9. 팀 규칙 리마인드 (딱 3개)¶

    1. 팀원: 지금처럼 만들고 공유방에 올리기만

    3. 전원 합의: 확정 기준은 facts(위키) — 개인 문서와 다르면 위키가 이김
