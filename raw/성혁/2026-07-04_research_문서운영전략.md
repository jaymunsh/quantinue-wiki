# FastAPI Agent Pipeline 문서 운영 전략

## 1. 현재 상황

현재 프로젝트는 하나의 FastAPI 기반 파이프라인을 중심으로 구성되어 있다.

여러 명이 파이프라인 중간에 위치한 각각의 Agent 또는 기능을 담당하고 있으며, 각 담당자는 본인이 맡은 영역을 중심으로 리서치하고 문서를 정리하고 있다.

팀원들은 개발 전문지식이 많지 않은 상태이며, 주로 AI를 활용해 HTML 형태의 문서를 생성하거나 갱신하고 있다.

현재 문서 작성 방식은 자유도가 높다는 장점이 있지만, 시간이 지날수록 다음과 같은 문제가 발생할 가능성이 크다.

```text
- 어떤 문서가 최신인지 알기 어려움
- 같은 내용이 여러 문서에 중복됨
- 확정된 내용과 단순 리서치가 섞임
- Agent 간 입력/출력 계약이 불명확해짐
- 전체 파이프라인 기준 문서가 사라짐
- 누가 언제 무엇을 바꿨는지 추적하기 어려움
- 문서가 많아질수록 SSOT가 무너짐
```

따라서 문서 작성 자체를 막기보다는, 각자의 자유로운 문서 작성을 유지하면서도 공식 문서 기준점을 따로 만드는 방식이 필요하다.

---

## 2. 핵심 원칙

이 프로젝트의 문서 운영 원칙은 다음과 같다.

```text
작성은 자유롭게, 공식화는 표준화한다.
```

즉, 각 팀원이 AI를 활용해 HTML, 메모, 리서치 문서 등을 자유롭게 만드는 것은 허용한다.

다만 공식 문서는 별도로 관리한다.

```text
팀원 HTML 문서 = 원본 자료 / 리서치 자료
MkDocs Markdown 문서 = 공식 문서 / SSOT
ADR 문서 = 결정 기록
```

중요한 것은 도구 자체가 아니라, 어떤 문서를 공식 기준으로 볼 것인지를 명확히 정하는 것이다.

---

## 3. 추천 솔루션: MkDocs Material

### 3.1 MkDocs Material이란?

MkDocs는 Markdown 파일을 기반으로 정적 문서 사이트를 생성하는 도구다.

Material for MkDocs는 MkDocs 위에서 동작하는 고급 테마이자 문서 프레임워크로, 다음 기능을 제공한다.

```text
- Markdown 기반 문서 작성
- 웹사이트 형태의 문서 제공
- 사이드바 / 탭 기반 내비게이션
- 검색 기능
- 다크 모드
- 코드 블록
- 표
- 콜아웃 / 경고 / 결정사항 표시
- 정적 사이트 배포
```

팀원 입장에서는 Markdown 문서만 작성하면 되고, 문서 담당자는 이를 기반으로 공식 문서 사이트를 만들 수 있다.

---

## 4. MkDocs Material을 사용하는 이유

현재 프로젝트에 MkDocs Material이 적합한 이유는 다음과 같다.

### 4.1 파일 기반이라 AI와 잘 맞음

MkDocs는 DB 기반 위키가 아니라 파일 기반 문서 시스템이다.

문서 원본은 보통 `docs/` 폴더 안의 `.md` 파일이고, 사이트 설정은 `mkdocs.yml`에서 관리한다.

따라서 Codex, Cursor, Claude Code, OpenCode 같은 AI 파일 에이전트와 함께 사용하기 좋다.

AI에게 다음과 같은 작업을 맡길 수 있다.

```text
- HTML 문서를 Markdown 공식 문서로 변환
- Agent 문서를 템플릿에 맞게 정리
- MkDocs 메뉴 구조 갱신
- 문서 간 입력/출력 불일치 검사
- ADR 후보 추출
- mkdocs build 실행 후 오류 확인
```

### 4.2 공식 문서와 원본 문서를 분리할 수 있음

팀원들이 만든 HTML을 그대로 공식 문서로 삼지 않고, 원본 자료로 보관한다.

그 후 문서 담당자가 AI를 활용해 공식 Markdown 문서로 정리한다.

```text
HTML = 자유 리서치 / 원본 자료
Markdown = 공식 문서
MkDocs = 공식 문서 사이트
```

### 4.3 검색 가능한 내부 지식베이스를 만들 수 있음

문서가 많아지면 사람들은 목차만 보고 찾지 않는다.

검색이 중요해진다.

MkDocs Material은 문서 사이트 내 검색 기능을 제공하므로, 다음과 같은 키워드로 문서를 찾을 수 있다.

```text
fallback
retry
agent input
json schema
error handling
FastAPI endpoint
confidence score
```

### 4.4 문서 구조를 강제할 수 있음

MkDocs는 `mkdocs.yml`의 `nav` 설정을 통해 공식 문서의 목차를 정할 수 있다.

따라서 공식 문서의 범위와 구조를 명확히 할 수 있다.

## 5. 추천 프로젝트 구조

문서 전용 저장소 또는 문서 전용 프로젝트를 만든다면 다음 구조를 추천한다.

```text
project-docs/
  AGENTS.md
  mkdocs.yml

  docs/
    index.md

    pipeline/
      overview.md
      data-flow.md
      api-contract.md
      error-handling.md

    agents/
      template.md
      agent-a.md
      agent-b.md
      agent-c.md

    decisions/
      index.md
      ADR-template.md
      ADR-001-agent-io-format.md

    changelog/
      2026-07.md

  incoming/
    agent-a/
      2026-07-03-agent-a.html
      2026-07-05-agent-a.html

    agent-b/
      2026-07-03-agent-b.html

  working/
    extraction-notes/
```

각 디렉터리의 역할은 다음과 같다.

| 경로                | 역할                |
| ----------------- | ----------------- |
| `docs/`           | MkDocs가 읽는 공식 문서  |
| `docs/pipeline/`  | 전체 파이프라인 문서       |
| `docs/agents/`    | Agent별 공식 문서      |
| `docs/decisions/` | ADR, 결정 기록        |
| `docs/changelog/` | 월별 변경 이력          |
| `incoming/`       | 팀원이 만든 HTML 원본 자료 |
| `working/`        | 문서 담당자의 중간 정리 자료  |
| `AGENTS.md`       | AI 에이전트에게 줄 작업 규칙 |
| `mkdocs.yml`      | MkDocs 사이트 설정     |

---

## 6. 문서 상태 구분

문서는 다음 세 단계로 관리한다.

```text
1. Raw / Incoming
   각자가 만든 HTML, 메모, 리서치 원본

2. Reviewed / 정리 중
   문서 담당자가 AI/Codex로 요약·분류·검토하는 단계

3. Official / SSOT
   MkDocs에 반영된 공식 문서
```

이 구분이 중요하다.

팀원이 만든 HTML을 곧바로 공식 문서로 인정하면 SSOT가 다시 무너질 수 있다.

---

## 7. 팀원들에게 최소한으로 요청할 사항

팀원들에게 Git, Markdown, MkDocs를 모두 배우게 하는 것은 현실적으로 부담이 크다.

따라서 팀원들에게는 복잡한 도구 사용을 요구하지 않고, HTML 문서 안에 최소한의 업데이트 정보를 남기게 한다.

### 7.1 업데이트 블록

팀원이 문서를 추가하거나 수정할 때마다 아래 블록을 남기도록 요청한다.

```text
[업데이트]
날짜:
작성자:
담당 Agent/기능:
상태: 리서치 / 제안 / 확정 / 폐기
변경 요약:
추가된 내용:
변경된 내용:
아직 결정이 필요한 내용:
다른 Agent나 전체 파이프라인에 영향이 있는 부분:
```

이 블록의 목적은 문서 담당자가 변경사항을 빠르게 파악할 수 있게 하는 것이다.

---

### 7.2 상태값

문서에 작성된 내용은 반드시 상태를 구분한다.

| 상태  | 의미                    |
| --- | --------------------- |
| 리서치 | 조사한 내용이며 적용 여부는 아직 모름 |
| 제안  | 적용 후보                 |
| 확정  | 팀에서 쓰기로 결정한 내용        |
| 폐기  | 검토했지만 사용하지 않기로 한 내용   |

상태값이 없으면 나중에 어떤 내용이 실제 결정사항인지 알기 어려워진다.

---

### 7.3 입력/출력 정보

Agent 프로젝트에서는 설명보다 입력/출력 계약이 중요하다.

각 Agent 담당자는 최소한 다음 정보를 작성해야 한다.

```text
이 Agent가 받는 것:
- 필드명:
- 타입:
- 예시:
- 필수 여부:

이 Agent가 내보내는 것:
- 필드명:
- 타입:
- 예시:
- 다음 Agent에서 쓰는지 여부:
```

---

### 7.4 기존 내용 수정 표시

같은 HTML 파일에 계속 내용을 추가하는 방식이라면, 변경 지점 표시가 매우 중요하다.

팀원들은 다음 구분을 남겨야 한다.

```text
추가: 새로 넣은 내용
변경: 기존 내용에서 바꾼 내용
삭제/폐기: 더 이상 쓰지 않는 내용
미정: 아직 결정 안 된 내용
```

---

## 8. 팀원에게 전달할 안내문 예시

```text
문서 정리 방식 관련해서 간단히 요청드립니다.

각자 기존처럼 AI를 활용해서 HTML 문서를 작성하셔도 됩니다.
다만 공식 문서로 정리할 때 변경 내용을 추적해야 해서, 앞으로 내용을 추가하거나 수정할 때는 아래 블록을 함께 남겨주세요.

[업데이트]
- 날짜:
- 작성자:
- 담당 Agent/기능:
- 상태: 리서치 / 제안 / 확정 / 폐기
- 변경 요약:
- 추가된 내용:
- 변경된 내용:
- 아직 결정이 필요한 내용:
- 다른 Agent나 전체 파이프라인에 영향이 있는 부분:

그리고 Agent 관련 내용은 가능하면 아래 항목을 구분해서 작성해주세요.

1. 이 Agent의 역할
2. 입력 데이터
3. 출력 데이터
4. 처리 흐름
5. 실패하거나 예외가 나는 경우
6. 다른 Agent와 연결되는 부분
7. 아직 미정인 부분

각자 작성하는 원본 HTML은 자유롭게 유지하고, 제가 MkDocs 기반 공식 문서로 정리하겠습니다.
공식 기준 문서는 나중에 MkDocs 문서 사이트에 반영된 내용을 기준으로 보겠습니다.
```

## 9. 같은 HTML 파일을 계속 갱신하는 경우

팀원들이 하나의 HTML 파일에 계속 내용을 추가한다면, 다음 문제가 생길 수 있다.

```text
- 무엇이 새로 추가됐는지 모름
- 기존 내용이 바뀌었는지 모름
- 오래된 내용이 최신처럼 보임
- AI가 매번 전체 문서를 다시 분석해야 함
- 중복 요약이 늘어남
- 확정과 제안이 섞임
```

따라서 같은 HTML을 계속 쓰더라도, 업데이트 구간을 명확히 나누게 해야 한다.

예시:

```html
<!-- UPDATE START: 2026-07-03 / 홍길동 / Search Agent -->
<section>
  <h2>2026-07-03 업데이트</h2>

  <h3>상태</h3>
  <p>제안</p>

  <h3>변경 요약</h3>
  <p>Search Agent 출력 포맷 후보 추가</p>

  <h3>추가 내용</h3>
  <p>...</p>

  <h3>결정 필요</h3>
  <p>confidence score 기준값 결정 필요</p>
</section>
<!-- UPDATE END: 2026-07-03 / 홍길동 / Search Agent -->
```

더 단순하게는 아래와 같은 구분선만 있어도 된다.

```text
==============================
업데이트: 2026-07-03 / 홍길동 / Search Agent
상태: 제안
==============================
```

핵심은 사람이든 AI든 변경 단위를 식별할 수 있어야 한다는 것이다.

## 10. 문서 담당자의 운영 워크플로우

문서 담당자는 다음 흐름으로 운영한다.

```text
팀원 HTML 수집
  ↓
incoming/에 날짜별 스냅샷 저장
  ↓
Codex 또는 AI로 변경사항 추출
  ↓
공식 문서 반영 후보 검토
  ↓
docs/agents/ 또는 docs/pipeline/ 갱신
  ↓
중요 결정은 ADR 후보로 분리
  ↓
mkdocs build로 문서 빌드 확인
  ↓
MkDocs 문서 사이트 반영
```

---

## 11. HTML 수집 방식

팀원들이 같은 HTML 파일을 계속 갱신하더라도, 문서 담당자 쪽에서는 날짜별 스냅샷으로 저장하는 것이 좋다.

예시:

```text
incoming/agent-a/2026-07-03-agent-a.html
incoming/agent-a/2026-07-05-agent-a.html
incoming/agent-a/2026-07-08-agent-a.html
```

이렇게 해야 이전 버전과 최신 버전을 비교할 수 있다.

## 12. Codex에게 맡길 수 있는 작업

### 12.1 HTML 변경사항 비교

```text
incoming/agent-a/2026-07-05-agent-a.html을 읽고,
이전 버전인 incoming/agent-a/2026-07-03-agent-a.html과 비교해줘.

다음 형식으로 정리해줘.

1. 새로 추가된 내용
2. 변경된 내용
3. 삭제되었거나 더 이상 언급되지 않는 내용
4. 공식 문서에 반영할 내용
5. 결정이 필요한 내용
6. 다른 Agent나 pipeline 문서에 영향 있는 내용

아직 파일은 수정하지 말고 먼저 보고만 해줘.
```

### 12.2 공식 문서 반영 후보 추출

```text
incoming/agent-a/2026-07-05-agent-a.html을 읽고,
공식 문서에 반영할 내용만 추려줘.

다음 기준으로 분류해줘.

1. Agent 역할
2. 입력 데이터
3. 출력 데이터
4. 처리 흐름
5. 실패 케이스
6. 다른 Agent에 영향 있는 내용
7. 결정이 필요한 내용
8. ADR 후보

아직 docs/ 파일은 수정하지 말고 보고만 해줘.
```

### 12.3 Agent 문서 갱신

```text
방금 정리한 반영 후보를 기준으로 docs/agents/agent-a.md를 갱신해줘.

조건:
- 기존 확정 내용은 삭제하지 말 것
- 중복 설명은 병합할 것
- 추정 내용은 검토 필요로 표시할 것
- 입력/출력은 표로 정리할 것
- 원본 HTML 경로를 참고 자료에 남길 것
```

### 12.4 전체 충돌 검사

```text
docs/agents/*.md와 docs/pipeline/*.md를 비교해서
입력/출력 필드명이 충돌하거나 누락된 부분을 찾아줘.

수정하지 말고 아래 형식으로 보고해줘.

- 문제 위치
- 충돌 내용
- 영향 받는 Agent
- 권장 수정 방향
```

### 12.5 ADR 후보 추출

```text
docs/agents/*.md에서 결정사항처럼 보이지만 ADR로 분리되지 않은 내용을 찾아줘.

특히 다음 항목을 우선 확인해줘.

- 입력/출력 포맷
- 에러 처리 정책
- 외부 API 의존성
- 파이프라인 순서
- 공통 데이터 구조
```

## 13. 공식 Agent 문서 템플릿

각 Agent 공식 문서는 아래 구조로 통일한다.

````
## 14. ADR 운영

ADR은 Architecture Decision Record의 약자이며, 중요한 결정과 그 이유를 기록하는 문서다.

리서치 문서는 시간이 지나면 낡지만, 결정 문서는 오래 남는다.

따라서 다음에 해당하는 내용은 ADR 후보로 분리한다.

```text
- Agent 간 입력/출력 포맷 변경
- 파이프라인 순서 변경
- 외부 API 또는 DB 의존성 추가
- 에러 처리 정책 변경
- 성능, 비용, 보안에 영향을 주는 결정
- 공통 데이터 구조 정의
````

---

## 15. ADR 템플릿

```markdown
# ADR-001: Agent 간 데이터 전달 포맷은 JSON으로 통일한다

## 상태

Proposed / Accepted / Deprecated

## 배경

여러 Agent가 서로 다른 형태의 데이터를 전달하면 파이프라인 통합 시 오류가 증가한다.

## 결정

Agent 간 데이터 전달은 JSON 객체로 통일한다.

## 대안

1. 자유 텍스트
2. Python dict 직접 전달
3. JSON Schema 기반 JSON

## 결과

- 각 Agent는 입력/출력 스키마를 문서화해야 한다.
- FastAPI endpoint와도 연결하기 쉬워진다.
- 단, 초기 작성 부담은 증가한다.

## 영향 범위

- Agent A
- Agent B
- Pipeline API
- Error Handling

## 관련 문서

- docs/pipeline/api-contract.md
- docs/agents/agent-a.md
```

## 16. AGENTS.md 예시

MkDocs 프로젝트 루트에 `AGENTS.md`를 두면, Codex 같은 AI 에이전트에게 문서 작성 규칙을 줄 수 있다.

````markdown
# AGENTS.md

이 저장소는 FastAPI Agent Pipeline의 공식 문서 저장소다.

## 문서 원칙

- `docs/` 아래 Markdown 문서만 공식 문서로 간주한다.
- `incoming/` 아래 HTML 파일은 원본 참고자료이며 공식 문서가 아니다.
- 팀원이 작성한 HTML 내용을 그대로 복사하지 말고, 공식 문서 템플릿에 맞춰 요약·정리한다.
- 확정된 내용, 제안, 리서치, 미정 사항을 반드시 구분한다.
- 기존 Accepted 상태의 결정사항은 임의로 삭제하거나 변경하지 않는다.
- 변경이 필요하면 먼저 "변경 제안"으로 작성한다.

## 공식 문서 반영 기준

공식 문서에는 다음 내용만 반영한다.

- Agent 역할
- 입력 데이터
- 출력 데이터
- 처리 흐름
- 실패 케이스
- 의존성
- 파이프라인 영향
- 확정된 결정
- 미정 사항

## ADR 작성 기준

다음에 해당하면 ADR 후보를 작성한다.

- Agent 간 입력/출력 포맷 변경
- 파이프라인 순서 변경
- 외부 API 또는 DB 의존성 추가
- 에러 처리 정책 변경
- 성능, 비용, 보안에 영향을 주는 결정

## 작성 스타일

- 한국어로 작성한다.
- 표를 적극적으로 사용한다.
- 추정은 "검토 필요"로 표시한다.
- JSON payload는 코드 블록으로 작성한다.
- 원본 HTML은 참고 자료에 경로만 남긴다.

## 검증

문서 수정 후 가능하면 다음 명령을 실행한다.

```bash
mkdocs build
````

빌드 오류가 있으면 원인을 설명하고 최소 수정한다.


## 18. 문서 담당자의 주간 운영 루틴

문서 담당자가 매일 모든 내용을 완벽하게 정리하려고 하면 병목이 생긴다.

따라서 주간 루틴을 정하는 것이 좋다.

### 매일 또는 작업 종료 시

```text
- 팀원들이 HTML 업데이트
- 업데이트 블록 작성
- 문서 담당자는 incoming/에 스냅샷 저장
```

### 주 2~3회

```text
- Codex로 변경사항 추출
- 공식 문서 반영 후보 정리
- 미정/충돌 항목 목록화
```

### 주 1회

```text
- 팀 전체 문서 리뷰
- 결정 필요한 항목 확정
- ADR 상태 변경: Proposed → Accepted
- MkDocs 사이트 최신화
```

문서 리뷰 때는 긴 리서치 문서를 읽기보다 아래 세 가지를 중심으로 확인한다.

```text
1. 이번 주 변경된 입력/출력
2. 결정이 필요한 사항
3. 다른 Agent에 영향 있는 변경
```

## 19. 우선 관리해야 할 문서

문서 담당자는 모든 문서를 같은 비중으로 관리하면 안 된다.

우선순위는 다음과 같다.

### 1순위: API / Agent Contract

각 Agent가 무엇을 받고 무엇을 내보내는지 정의한다.

```text
입력 데이터
출력 데이터
필수 필드
타입
예시
다음 Agent 사용 여부
```

이 문서가 깨지면 파이프라인이 깨진다.

### 2순위: Pipeline Data Flow

전체 Agent 실행 순서와 데이터 흐름을 정리한다.

```text
User Input
  → Agent A
  → Agent B
  → Agent C
  → Final Output
```

### 3순위: ADR

왜 그렇게 결정했는지를 기록한다.

나중에 의견이 충돌하거나 팀원이 바뀌었을 때 ADR이 기준이 된다.

### 4순위: Research

긴 리서치 문서는 공식 문서에 전부 넣지 않는다.

공식 문서에는 결론, 영향, 결정 필요 사항만 반영한다.

---

## 20. 공식 문서에 넣을 내용과 넣지 않을 내용

### 공식 문서에 넣을 내용

```text
- Agent 역할
- 입력 데이터
- 출력 데이터
- 처리 흐름
- 실패 케이스
- 의존성
- 확정된 결정
- 미정 사항
- 다른 Agent와의 연결점
- 파이프라인에 미치는 영향
```

### 공식 문서에 과하게 넣지 않을 내용

```text
- 긴 리서치 전문
- AI가 생성한 장황한 설명
- 비슷한 내용의 반복
- 근거 없는 추측
- HTML 스타일 코드
- 개인적인 메모
- 아직 검증 안 된 외부 자료 요약
```

공식 문서는 많은 정보를 담는 것이 목적이 아니라, 팀이 판단하고 실행할 수 있는 기준을 제공하는 것이 목적이다.

## 21. 피해야 할 운영 방식

### 21.1 팀원 모두에게 Git을 강제하지 않기

초기에는 문서 담당자만 Git을 관리하는 것이 좋다.

```text
초기:
문서 담당자만 Git 관리

중간:
팀원은 GitHub 웹에서 Markdown 수정

이후:
각자 브랜치 / PR 기반 운영
```

처음부터 Git CLI, 브랜치, PR, conflict resolution을 강제하면 문서 시스템이 아니라 Git 교육 프로젝트가 될 수 있다.

---

### 21.2 HTML을 공식 문서로 인정하지 않기

HTML은 원본 자료로만 둔다.

HTML을 공식 문서로 인정하면 다음 문제가 생긴다.

```text
- 스타일과 내용이 섞임
- diff 확인이 어려움
- 검색 품질이 떨어짐
- 중복 내용이 쌓임
- AI가 편집하기 어려움
- MkDocs 구조와 따로 놀게 됨
```

---

### 21.3 모든 내용을 MkDocs에 넣지 않기

공식 문서는 짧고 명확해야 한다.

각 Agent 공식 문서는 처음에는 다음 정도를 목표로 한다.

```text
짧으면 1~2페이지
길어도 3~5페이지
```

긴 리서치는 `incoming/` 또는 `research/`에 두고, 공식 문서에는 결론만 넣는다.

---

### 21.4 결정사항을 일반 문장으로 묻어두지 않기

예를 들어 아래 문장은 위험하다.

```text
결과는 JSON으로 넘기면 될 것 같다.
```

이 문장은 단순 설명이 아니라 결정 후보다.

따라서 다음처럼 상태를 붙여야 한다.

```text
상태: Proposed
결정 필요: Agent 간 데이터 전달 포맷을 JSON으로 통일할지 여부
```

확정되면 ADR로 올린다.

## 22. 최종 운영 모델

현재 프로젝트 상황에서는 다음 모델이 가장 현실적이다.

```text
팀원:
  자유롭게 HTML 작성
  업데이트 블록만 남김

문서 담당자:
  HTML 스냅샷 보관
  Codex로 변경사항 추출
  MkDocs 공식 문서 반영
  ADR 후보 분리
  주 1회 결정사항 리뷰

공식 기준:
  MkDocs에 반영된 문서만 SSOT
```

---

## 23. 핵심 요약

이 문서 운영의 핵심은 다음과 같다.

```text
1. 팀원의 자유로운 문서 작성을 막지 않는다.
2. HTML은 원본 자료로만 본다.
3. 공식 문서는 MkDocs Markdown으로 관리한다.
4. 각 Agent 문서는 템플릿을 통일한다.
5. 입력/출력 계약을 최우선으로 관리한다.
6. 중요한 결정은 ADR로 분리한다.
7. Codex 같은 AI 파일 에이전트를 문서 정리 보조자로 활용한다.
8. 팀원에게는 업데이트 블록만 최소 요청한다.
9. 문서 담당자는 날짜별 HTML 스냅샷을 보관한다.
10. MkDocs에 반영된 문서만 SSOT로 본다.
```

가장 중요한 원칙은 다음 한 문장으로 정리할 수 있다.

```text
팀원에게는 잘 정리된 최종 문서를 요구하지 말고,
문서 담당자가 공식화할 수 있는 변경 단위를 받는 구조를 만든다.
```
