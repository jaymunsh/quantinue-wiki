    코드 안건 당사자
    B3 창욱 최종필드 ↔ 은미 v3.0 소비 충돌 3종 — 되살리기 vs 게이트 재설계 택1 창욱·은미
    B5 tb_critic_verdict 필드 확정 — 지현 초안 → 미연 확정만 미연

    C1 MVP 성공 기준 정의 (멘토 최우선) 멘토·전원
    B3 상세 — 충돌 3필드

    은미 v3 게이트가 창욱이 삭제 확정한 필드를 아직 소비 → 이대로 실행하면 payload가 깨진다. 지현 마스터 ERD(07-09)가 "택1"을 게이트 재설계 + 축 정리로 정리:

    필드 창욱 최종 은미 v3(소비) 07-09 방향
    confirmed_score ❌ 삭제 확정(07-08) 뉴스 게이트 ≥0.50 + 보너스 ×0.03 되살리기 아님 → source_trust/grade로 게이트 재설계
    cross_source_confirmed ❌ 제거(disclosure_ref 대체) v3 문서에 아직 남음 v3에서 실제로 뺄지 재확인
    source_trust ↔ grade_score 유지 — 다른 축(grade=코드·사전 / trust=LLM·사후) "중복 → 빼달라" 축 정의 합의(둘 다 vs grade만)

    경과(07-09 지현 마스터): confirmed_score·cross_source는 창욱 ERD에서 물리 삭제됨 → 은미 게이트 재설계가 필수 · 같은 삭제 필드를 Critic payload도 받아 하드룰 깨짐(B5).

    경과(6차 회의 07-09): 뉴스 대표기사 선정 기준에서 막힘 — 창욱(실내가중치) ↔ 은미 파일 방식 불일치, "대표" 정의부터 재정리 필요 → 창욱 재작성 후 다음 회의. 센티멘트는 라벨 빼고 sentiment_score 하나로 통합(0~1 소수점) 방향. → 회의록

    B5 상세 — tb_critic_verdict 초안

    지현이 미연 인수인계 v1.2로 정리: id·signal_id·cycle_id·decision(pass/reject)·agree·category(weak_evidence·late_entry·pump·event_risk·macro_riskoff·halted)·objection·confidence·decided_layer(hard_rule/llm/gate).

    경과(5차 07-08): agree↔decision 중복 · objection_category · decided_layer 용어 확인 중.

    경과(07-09 지현 마스터): Critic payload가 삭제 확정 필드(confirmed_score·cross_source_confirmed)를 아직 받음 → 하드룰(grade≥0.6 등)·시각 컬럼이 스키마와 어긋나 신선도·선반영 오판 위험. B3 정합과 함께 처리.

    경과(6차 회의 07-09): 미연이 4계층 발표 — 정합성 체크(관문)·리스크오프 risk_score≥70·고점추격 ret_5d≥15% AND rsi≥70·실시간 시세=은미 스냅샷 그대로 사용 방향. 미검증 각도=뉴스 grade_score("봉투")로 판정. → 회의록

    B8 상세 — tb_review 한 그림 (schema keeper 제안)

    tb_review 스키마는 데이터계약 반영 완료(#9). 테이블을 늘리지 않고 JOIN 하나로 은미 스펙을 전부 충족 — tb_memory_entries(임시 이름) 폐기.

    B9 필드계약 표 접미사 _signals 통일 (tb_technical ↔ tb_technical_signals) 지현·은미
    B10 필드계약 cycle_id ↔ trade_date 키 매핑 — 생성 주체=오케스트레이터(문서화) 지현·은미
    B11 필드계약 리뷰어 채점·운영 세부 (기준가·T+5·메모리 주입·NO_TRADE) 성혁·은미·지현
    B12 필드계약 updated_at = 상태 변경 테이블만 (#10) — tb_account·tb_order 지현·성혁
    B13 필드계약 decision_close 신설 — 보류 P0인데 v3.0에도 없음 → 재확인 은미·성혁
    B14 필드계약 실시간 스냅샷 형식(개별 컬럼 vs JSONB) + 손절/매도 필드 제거 은미·지현
    B4 필드계약 기술 신호 값 형태 (trend 문자열·macd 숫자) 지현·은미
    B6 필드계약 PM sizing_hint 형식 (suggested_weight ↔ 25%·5종목·−15%) 은미·지현
    A5 인프라 cycle_id, 애초에 필요한가? — 유지 중이나 필요성 미결 지현·은미
    A6 인프라 LLM 모델명 확정 (분석가=mini · Strategist=상위) 전원
    A7 인프라 ml_prob_up 1차 사용? (02 빈값 {} ↔ 07 POLICY ≥0.50) 지현·은미
    C3 멘토 화면·사용자 흐름 산출물 멘토
    C4 멘토 간트 구현단계 세분화 + PPT 전날 완성 멘토
    B11 상세 — 리뷰어 채점 4항

    ① 기준가 P0: 매수=체결가 / 보류=판단일 종가 · ② 채점 확정=T+5(창욱 백테스트 +1/3/5d와 동일 창) · ③ 1차 메모리 주입 허용 여부 — MVP "observe-only(기록만)"와 은미 STEP4(주입)가 갈림, 성혁 "참고 제공은 허용" 제안 · ④ NO_TRADE 회고 → 4차 회의: 50종목 전량 보류 포함 기록으로 결론(확정 반영 대기).

    A5 상세 — cycle_id 필요성 논증

    기존 설계에 있다고 정답은 아님. 하류는 signal_id FK 체인으로 이미 이어지고, 상류는 자기 시간 키로 살며, 한 실행의 묶음·시각은 tb_strategist_signals.created_at으로 도출 가능. 그럼 cycle_id가 실제로 하는 일이 남나? 필요하면 "무엇을 위해"부터 정의하고 최소 도입 (없애면 A3·B10도 소멸).

    경과(07-08): 지현 명세는 cycle_id 유지·생성 주체=오케스트레이터로 문서화 — "필요한가"는 여전히 미결.
