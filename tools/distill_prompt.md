# 증류 작업 지시서 (distill_prompt) v1

> 이 파일은 클로드 코드에게 주는 작업 명세서다. 운영자: "이 지시서 읽고 증류 돌려줘" 한 마디로 실행.
> 결과가 이상하면 이 지시서에 규칙을 한 줄씩 추가해서 키운다 (프롬프트 = 코드).

## 입력

- `raw/*/` 안의 파일 중 **지난 증류 이후 추가·변경된 것만** 처리
- 지난 증류 시점·회차는 `documents/STATUS.md` 맨 위 "기준" 줄 참조
- HTML이 크면 파이썬으로 텍스트만 추출 후 분석 (스타일·스크립트 제거)
- 문서 맨 위 `[업데이트]` 3줄 헤더가 있으면 그 부분을 우선 정밀 분석, 없으면 전체 대조

## 1단계 · 주장(claim) 추출

문서에서 "팀의 결정·설계에 해당하는 문장"을 주장 단위로 뽑는다.

주장 하나의 형식:
```yaml
- 내용: "손절 기준을 -7% 고정에서 ATR 배수 방식으로 변경"
  주제: 데이터계약        # documents/facts/의 어느 파일에 속하나 (파이프라인/데이터계약/일정/화면/결정로그)
  상태: 제안              # 리서치 / 제안 / 확정후보
  작성자: 지현            # raw/ 폴더명
  출처: raw/지현/2026-07-08.html
  결정필요: "ATR 배수값 (1.5? 2.0?)"   # 없으면 null
```

**굵기 기준 (중요):**
- ✅ 좋은 굵기: "DB는 Postgres+TimescaleDB로 한다" / "후보는 5~10개로 제한"
- ❌ 너무 잘게: "PER은 주가수익비율이다" (용어 설명은 주장 아님)
- ❌ 너무 크게: "시스템 아키텍처 전체" (문서 통째는 주장 아님)
- 배경 설명·교육용 내용·비유는 건너뛴다. **결정·수치·계약·일정만** 주장이다.

## 2단계 · 대조 (4분류)

각 주장을 `documents/facts/*` (확정)와 `documents/질문.md` (기존 미정)과 비교:

| 분류 | 기준 | 처리 |
|---|---|---|
| ♻️ 이미 있음 | facts 또는 기존 안건과 실질 동일 | 버림. 개수만 집계 |
| 🆕 새것 | 처음 등장 | documents/질문.md 또는 STATUS 미정란에 추가 |
| 🔄 달라짐 | 같은 주제인데 내용이 기존과 다름 | "기존 A → 이번 B" 형태로 안건화 |
| ⚠️ 충돌 | **다른 작성자**의 주장과 어긋남 | 최우선 — STATUS 충돌란 + 안건 |

- "실질 동일" 판단은 표현이 아니라 의미 기준 (숫자·방식이 같으면 동일)
- 멘토 피드백의 지적·요구는 자동으로 🆕 안건 취급 (우선순위 높음 표시)

## 3단계 · 산출

1. **documents/STATUS.md 전체 덮어쓰기** — 형식:
   - 맨 위: `> 기준: YYYY-MM-DD · 증류 N회차 · 이번에 처리한 파일 목록`
   - 카운트: ✅확정 n / ❓미정 n / ⚠️충돌 n
   - ⚠️ 충돌 상세 (있으면 맨 위에)
   - ✅ 최근 확정 5개 (출처 포함)
   - 파트별 한 줄 근황 (지현/창욱/은미/미연/성혁/멘토)
2. **documents/질문.md에 추가** — 기존 항목 유지, 제목이 유사하면 병합(중복 금지). 각 안건: 안건명 · 왜 정해야 하나 · 제기(출처)
3. **documents/업데이트로그.md 맨 위에 1줄 추가** — `| 날짜 | 🌀 반영 | 처리 문서·핵심 변화 한 줄 |`
4. **documents/추천제안.md 갱신** (이번 증류에 의미 있는 델타가 있을 때만) — 아래 "추천제안 갱신 루브릭"대로 상위 3~5개를 재작성하고 헤더 "최종 갱신" 날짜·근거를 고친다. **facts는 안 건드린다.**
5. **채팅 보고** — `새 주장 N건 · 충돌 N건 · 중복 소멸 N건 · 추천 안건 순위`

### 추천제안 갱신 루브릭 (documents/추천제안.md)

이 페이지는 **손글씨가 아니라 산출물** — 매 증류가 재생성한다. 상위 3~5개를 이 기준으로 뽑는다:

1. **지금 릴레이를 막는 병목 1개**를 최상단에
2. **이번 증류에서 새로 생긴 리스크·기회**
3. **저비용·고효과 순** 정렬
4. **전에 제안했으나 아직 안 된 것** 재부상 (사라지지 않게 — loop-until-done)
5. 모든 제안에 **근거 링크**(회의록·안건·facts)

- 이 페이지는 **제안**이다 — 확정처럼 쓰지 말고 3인칭 중립으로. 결정은 팀·병입이 한다.
- 델타가 없으면 헤더 날짜만 갱신하고 내용은 유지 (불필요한 재작성 금지).
- **재생성 범위 주의**: 매번 새로 뽑는 건 상단 "지금 우선순위/병목" 부분이다. **상세 도메인 제안(예: 뉴스 구성 방식 · mermaid 포함)은 보존**하고, 그 주제의 결정이 실제로 바뀌었을 때만 손본다 — 힘들여 쓴 제안을 매 증류마다 지우지 말 것.

## 회의 전사 처리 (raw/회의/* — 특별 규칙)

`raw/회의/`의 파일(녹음 전사 txt 등)은 일반 문서와 달리 **결정의 원천**이다. 주장 추출 대신 아래로 처리한다:

1. **`documents/회의록/YYYY-MM-DD.md` 생성** — 아래 템플릿 그대로. 날짜는 파일명에서(같은 날 2건이면 `-2`). `documents/회의록/index.md` 목록 맨 위에 1줄 추가.
2. **결정 감지** — 합의·확정된 발언을 회의록 "결정" 절에 나열하되 **facts에는 쓰지 않는다.** 보고 마지막에 `병입 대기 N건`으로 제시 → 운영자가 "병입해줘" 하면 그때 결정로그·facts 반영.
3. **보류·미결** — documents/질문.md의 해당 안건에 "논의 경과" 한 줄 갱신 (안건 제거는 병입 시).
4. **액션 아이템** — 회의록에만 기록. 담당·기한 필수(없으면 `기한 미정`으로 명시).
5. 전사 특성상: 오인식 추정 단어는 `[?]` 표시, 발언자 불명은 `(불명)`. **지어내지 않는다** — 전사에 없는 결정을 만들면 안 됨(거짓 확정이 최악).

### 회의록 템플릿

```markdown
# 🗣️ 회의록 YYYY-MM-DD

!!! abstract "3줄 요약"
    (무엇이 정해졌고, 무엇이 남았고, 다음 스텝은)

## ✅ 결정된 것 (병입 대기 → 확정되면 결정 로그 # 부여)
| 결정 | 관련 안건 | 근거 발언 요지 |
|---|---|---|

## ⏸️ 보류·계속 논의
| 안건 | 경과·이유 |
|---|---|

## 📋 액션 아이템
| 담당 | 할 일 | 기한 |
|---|---|---|

## 💬 논의 맥락 (선택 · 왜 그렇게 결정했는지 짧게)
```

## 산출물 작성 스타일 (모든 md 공통)

- **쉬운 말 우선** — 어려운 용어가 나오면 바로 옆에 한 줄 풀이 (예: "멱등성(같은 걸 두 번 해도 결과가 한 번)")
- 긴 설명보다 **표·목록** — 나열은 표로, 흐름은 번호 목록으로
- **흐름·구조·관계는 Mermaid 다이어그램 적극 활용** (MkDocs Material에서 자동 렌더). 아래 상황이면 글 대신/글과 함께 그림:
  - 단계 흐름(A→B→C) · 분기(허용/차단) · 피드백 루프 → `flowchart`
  - 테이블 관계·스키마 → `erDiagram`
  - 순서(누가 누구에게 무엇을 넘기나) → `sequenceDiagram`
  - 상태 변화(초안→확정→폐기) → `stateDiagram-v2`
  - 일정 → `gantt`
  - 코드블록은 ```mermaid 펜스로. 노드 라벨엔 담당·타입을 `<br/><i>지현·규칙</i>` 식으로. 색 구분(LLM/코드 등)이 이해를 도우면 classDef 사용.
- 항목마다 **왜 중요한지 한 줄** 을 붙일 수 있으면 붙인다 (상세가 필요한 곳만, 장황함 금지)
- 읽는 사람 = 비개발자 팀원 기준. "이 문장을 팀원이 처음 읽고 이해하는가"로 자체 검사

### 중복·언어 규칙 (2026-07-06 추가 — 위키 난잡화 방지)

- **한 사실은 한 곳에만**: 열린 안건 상세 = `documents/질문.md`에만 · 확정 이력 = `documents/facts/결정로그.md`에만. 다른 문서에 같은 목록을 복붙하지 말고 "→ 회의 안건 참조" 한 줄 링크로 가리킨다.
- **독자 분리**: 팀 페이지(홈·facts·질문·산출물)에서는 증류·병입·회차 같은 운영 용어 금지 — "확정"으로 표현. 운영 용어는 STATUS·가이드·시스템설명서에서만.
- **안건 코드(A1·B8 등)는 질문.md 안에서만** 정의·나열. 다른 페이지에선 코드 대신 짧은 이름+링크 (표 안의 참조 셀만 예외).
- **문체 = 3인칭 중립**: "성혁 담당"처럼 쓴다. "나"·"내" 금지 (팀원 모두가 읽는 문서).
- 페이지 머리 요약은 admonition 박스(`!!! success/abstract/warning`) 1~2개 — 여러 줄 `>` 인용문 벽글 금지.

## 금지 (절대 규칙)

- **documents/facts/ 를 직접 수정하지 마라.** 병입은 운영자의 별도 지시("○번 확정 병입해줘")로만 한다.
- **원문에 없는 내용을 지어내지 마라.** 모든 주장에 출처 필수. 출처를 못 대면 그 주장은 버린다.
- 애매하면 "확정후보"가 아니라 "제안"으로 분류하라. (거짓 확정이 최악)
- documents/질문.md의 기존 안건을 삭제하지 마라 (해결된 안건은 운영자가 병입 시 정리).

## 병입 지시를 받았을 때 (별도 명령)

운영자가 "안건 N번 ○○로 확정, 병입해줘" 하면:
1. `documents/facts/해당주제.md`에 확정 내용 추가 — 형식: `- 내용 (확정 YYYY-MM-DD, 출처: ..., 결정 #k)`
2. `documents/facts/결정로그.md`에 1줄: `| #k | 날짜 | 결정 | 이유 한 줄 |`
3. documents/질문.md에서 해당 안건 **행을 삭제**(취소선만 남기지 말 것)하고, 하단 `## ✅ 최근 종결 안건`의 접힌 표(`??? note`)에 `| 코드 | 한 줄 | ✅ 결정 #k |` 1줄 추가. documents/STATUS.md 갱신
4. documents/업데이트로그.md 맨 위에 1줄 추가 — `| 날짜 | ✅ 확정 | 결정 내용 한 줄 (#k) |`
5. 뒤집힌 결정은 삭제하지 말고 결정로그에 새 번호로 "변경" 기록
