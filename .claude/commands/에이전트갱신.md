## 매핑표 (이름 → 위키 페이지 · repo 브랜치)

| 이름 | 위키 페이지 | 브랜치 (예상) |
|---|---|---|
| reviewer | documents/에이전트/reviewer.md | reviewer-agent |
| critic | documents/에이전트/critic.md | critic-agent-1 |

| info | documents/에이전트/info.md | collector 계열 — `git branch -a`로 확인 |
| core | documents/에이전트/core.md | 여러 브랜치 산재 — 지현 브랜치 확인 |

- repo 경로: `~/Documents/ClaudeCode/quantinue/quantinue`
- 브랜치가 예상과 다르면 `git -C <repo> branch -a` 로 실제 목록을 확인하고 가장 최신·완성된 브랜치를 고른 뒤 보고에 명시.

## 절차

1. **읽기 전용 원칙** — 그 repo의 워킹 브랜치를 바꾸지 않는다. 파일은 반드시
   `git -C <repo> show <브랜치>:<경로>` 로 읽는다 (checkout·커밋·push 금지).
2. 참조 커밋 확인: `git -C <repo> rev-parse --short <브랜치>` → 페이지 상단 admonition의
   `참조: <브랜치> @ <커밋>` 을 갱신. **커밋이 기존과 같으면 "변경 없음"으로 보고하고 끝**.
3. 페이지 갱신 — 기존 템플릿(역할 → 동작 흐름 → 코드 구조 → 사용 스키마 → 핵심 규칙 → 계약·결정) 유지:
   - 흐름·핵심 위주. 함수 시그니처·라인 단위 금지 (코드 바뀌면 낡음 — repo가 진실, 위키는 이해용)
   - 흐름은 Mermaid, 파일 역할은 표
   - stub 페이지(strategist·info·core)가 처음 채워지는 경우: critic.md 를 템플릿 A로 삼는다
   - 위키 baseline(파이프라인·데이터계약)과 코드가 어긋나면 **코드를 따르지 말고** "정합 필요" warning 박스로 표기
4. `documents/에이전트/index.md` 의 구현 상태 표 갱신 (⚪ 설계 예정 → ✅ 구현 등).
5. `documents/업데이트로그.md` 맨 위에 1줄 추가 (`🔄 변경`).
6. `python3 tools/lint.py && python3 -m mkdocs build` 로 검증 후 보고:
   `페이지 | 브랜치@커밋(전→후) | 주요 변경점 | 정합 이슈`
