# 🔌 데이터 계약 (통합 인터페이스)

> **상태: 🟡 초안(제안) — 회의 확정 전.** schema keeper(성혁)가 은미·창욱·설계서·solutions의 스키마를 하나로 맞춘 제안. 회의에서 ⚠️만 결정하면 확정된다.
> **왜 이게 제일 중요한가:** 파이프라인은 "각 파트가 데이터를 주고받아" 돈다. 필드 이름 하나·타입 하나만 어긋나도 SELECT/파싱이 깨져 전체가 멈춘다. 이 문서가 그 "주고받기 규칙(계약)"이다.

---

## 1. 한눈에 — 데이터가 흐르는 법 (Pull 방식)

```
각 팀이 자기 신호를 DB에 저장(INSERT)  →  Strategist가 필요한 것만 읽음(SELECT)
[창욱] 공시·뉴스 ┐
[지현] 기술·매크로 ┼─▶ (같은 cycle_id + ticker로 묶어) ─▶ [은미] Strategist 판단
[성혁] 과거회고 ┘                                                 │
                                                    tb_strategist_signals 저장
                                                                 │
                                              [미연] Critic → [지현] PM·게이트·실행
```
- **Pull(우편함) 방식**: 각 팀은 "저장"만, 읽을 필드 선택은 Strategist가 함. (은미 확정)
- 모든 신호는 **`cycle_id` + `ticker`** 두 열쇠로 "같은 판단 회차의 같은 종목"끼리 매칭.

---

## 2. 공통 규칙 (⚠️ = 회의 결정 필요, → 뒤는 제안 기본값)

| 항목 | 현재 실태 | 제안 (기본값) | 안건 |
|---|---|---|---|
| **판단 회차 키** | 은미 `cycle_id` / solutions `run_id` / 창욱 없음 | → **`cycle_id`** (BIGINT) 로 통일. 08:30 실행 때 **오케스트레이터가 1개 발급** | A3 |
| **테이블명 접두사** | 은미 `tb_` / 창욱 없음 | → **`tb_` 붙이기**(namespacing). 창욱 코드는 rename 또는 view 1줄 | B1 |
| **종목·날짜 공통키** | 공통 | `ticker` VARCHAR · `as_of` DATE (매크로만 ticker 없음 = 시장 전체) | — |
| **점수 스케일** | 신호=0~1 / 확신=0~10 혼재 | → **신호 점수 0~1 · conviction·문턱 0~10** 고정(서로 다름 주의) | — |
| **매크로 risk_score 범위** ⚠️ | 은미 0~10 가정 / 설계서 0~100 / 창욱 0~1 | → **0~100** (설계서 가중식이 0~100) · 은미 거부권 문턱을 60/80으로 | A4 |
| **저장소** ⚠️ | 창욱 SQLite(코드有) / 은미·설계서 Postgres | → **개발 로컬 SQLite → 통합 시 Postgres**. 스키마 동일 유지(SQLite↔PG 호환 타입만) | A2 |
| **투자유형** ⚠️ | 공격1 / 안정+공격2 / 균형 3자 불일치 | → 회의 결정 필수. 계약은 `inv_type`으로 파라미터화(어느 유형이든 이 계약은 유지) | A1 |
| **LLM 모델** ⚠️ | GPT-4o / gpt-4o-mini / GPT-5.4mini·5.5 | → 분석가=싼 모델(mini급) · 결정자(Strategist)=상위. 정확한 모델명 확정 | 충돌7 |
| **전송 형식** | 공통 | JSON (`model_dump_json()`) = DB row 한 줄. Pydantic 스키마로 강제 | — |

## 3. 신호 테이블 계약 — "누가 무엇을 채우나"

> 각 파트는 이 표의 필드만 채우면 된다. ✅=Strategist가 판단에 씀 · 📄=GPT 참고만 · 🟢합의가능 · ⚠️결정필요

### 📄 tb_disclosure_signals — 공시 (소유: 창욱)
| 필드 | 타입 | 용도 | 상태 |
|---|---|---|---|
| ticker · cycle_id · as_of | VARCHAR·BIGINT·DATE | 공통키 | 🟢 |
| has_signal | SMALLINT | 오늘 공시 유무(0이면 스킵) | 🟢 |
| hard_block ✅ | SMALLINT | 1=즉시 보류(상폐·거래정지 등 객관사실만) | 🟢 |
| sentiment_score ✅ | NUMERIC(0~1) | 호재/악재 | 🟢 |
| importance ✅ | NUMERIC(0~1) | 사건 크기 | 🟢 |
| event_type 📄 | VARCHAR | 11종 온톨로지(§5) | 🟢 |
| reason 📄 | TEXT | 한 줄 근거 | 🟢 |

> 창욱 DisclosureBundle은 24필드지만 **Strategist가 SELECT하는 건 위 7개.** 나머지(confidence·ref·risk_score 등)는 저장은 하되 계약 밖(감사·GPT참고용).

### 📰 tb_news_signals — 뉴스 (소유: 창욱)
| 필드 | 타입 | 용도 | 상태 |
|---|---|---|---|
| ticker · cycle_id · as_of | | 공통키 | 🟢 |
| has_signal · hard_block ✅ | SMALLINT | 유무 · 즉시보류 | 🟢 |
| sentiment_score ✅ | NUMERIC(0~1) | 호재/악재 | 🟢 |
| grade_score ✅ | NUMERIC(0~1) | 출처 등급(ALLOW1.0/GRAY0.6/BLOCK0.0) | 🟢 |
| confirmed_score ✅ | NUMERIC(0~1) | 확정 비율(=confirmed/count) | 🟢 |
| **cross_source_confirmed** ✅ | SMALLINT | 2곳+ 교차확인=1 → 확신 +0.15 | ⚠️ **창욱 Bundle에 없음 → 추가 필요 (B3)** |
| top_evidence · reason 📄 | JSONB·TEXT | 대표 헤드라인 · 근거 | 🟢 |

### 📈 tb_technical_signals — 기술 (소유: 지현)
| 필드 | 타입 | 용도 | 상태 |
|---|---|---|---|

| rsi · macd 📄 | NUMERIC·VARCHAR | 참고지표 | ⚠️ macd 라벨 vs 숫자 (B4) |

|---|---|---|---|

## 4. 🔗 나(성혁)의 계약 — tb_memory_entries (Reviewer 출력 → Strategist 입력)

> **이게 내 Reviewer 파트의 인터페이스다.** 은미가 이 스키마를 확정해 소비함: ticker별 최근 5개를 GPT 프롬프트에 주입 → "같은 실수 반복 방지". 내 회고 출력을 여기 맞춰야 한다.

| 필드 | 타입 | 내가 채우는 내용 |
|---|---|---|
| id | BIGSERIAL | 자동 번호 |
| ticker · cycle_id | VARCHAR·BIGINT | 어느 종목·어느 회차의 회고인가 |
| side | VARCHAR | 그때 판단(매수/보류) |
| conviction | NUMERIC | 그때 확신 |
| **outcome** | NUMERIC | 결과(수익률 등) ← **내 Scorer가 코드로 채점**(내 초안의 metric_summary) |
| **lesson** | TEXT | 교훈 한 줄 ← **내 Reflector가 LLM으로 생성**(내 초안의 lesson) |
| created_at | TIMESTAMPTZ | 저장 시각 |

> 📌 내 `Reviewer_회고에이전트_초안.md`의 review_note(metric_summary·lesson)를 이 7컬럼에 매핑하면 됨. **1차 observe-only라도 이 테이블은 채운다**(전략 반영만 2차). → 안건 **B7**.

## 5. 공유 온톨로지 — event_type 11종 (공시·뉴스 공통, 소유: 창욱)

`earnings · guidance_change · ma · capital_raise · management_change · insider_trade · product_deal · analyst_rating · regulation_legal · delisting_halt · other`

> 하드리스크(거래정지·파산·상폐·회계문제·희석)는 event_type과 별개로 **hard_block=1**로 강제(창욱 정책). Strategist·게이트가 이걸 최우선 존중.

## 7. 이 문서가 강제하는 결정 (회의 체크리스트)

**A. 파이프라인 차단급 (먼저):** A1 투자유형 · A2 저장소 · A3 cycle_id · A4 risk_score 범위
**B. 필드 계약 (내가 표준화 제안):** B1 tb_ 접두사 · B2 category→sector · B3 cross_source_confirmed · B4 기술 값형태 · B5 Critic payload · B6 sizing_hint · B7 memory_entries
**+ 추가 발견:** ml_prob_up을 1차 기술 카운트에 쓸지(설계서 ML 미연결과 충돌)

> ✅ 이 ⚠️들만 회의에서 정하면 → 이 문서가 **확정 데이터 계약(SSOT)** 이 되고, 5명이 이걸 보고 병렬로 구현하면 통합이 안 깨진다. (병입 시 🟡초안 → 🟢확정)
