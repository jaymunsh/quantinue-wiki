## A. `/계약대조` — 스키마 drift 검사 🎯 최우선

**문제** — schema keeper의 핵심 리스크: 위키 데이터계약(14테이블)은 확정인데, 팀원들이 각자 브랜치에서 구현하며 코드 스키마가 조용히 어긋남. 실제 사례: `news_signals` vs `tb_news_signals`(→결정 #1), 은미 스키마의 매도/손절 필드(→4차-2 회의), decision_close 누락(B13).

**계획**
1. `.claude/commands/계약대조.md` 신설 — /에이전트갱신과 같은 패턴
2. 절차: quantinue repo 각 브랜치의 `schemas.py`·`core/schemas.py`를 `git show`로 읽기전용 조회 → 데이터계약.md 테이블·필드와 대조
3. 출력: `테이블 | 위키에만 | 코드에만 | 타입 불일치 | 처방` 표 + 안건 후보 제안 (facts 직접 수정 금지 — 발견만)
4. 매핑(브랜치→스키마 파일 경로)은 명령 파일 상단 표로 — /에이전트갱신처럼 수정 지점 한 곳

- 공수: 명령 파일 1개 (~30분)
- 시점: **Gate2(7/13) 전후 구현 본격화 때** — 그 뒤 매주 회의 전 1회 실행
- 성공 기준: B9(접미사)·B10(키 매핑)·B13(decision_close) 같은 안건이 회의 전에 자동으로 표로 나옴

## B. raw 삭제 방지 훅 🛡️

**문제** — 불변식 #1("raw는 불변")이 문서 규칙일 뿐. 실수 삭제 사례 이미 1회(07-06, git checkout으로 복구).

**계획**
1. `tools/hooks/pre-commit` 작성: 스테이징에 `raw/` 삭제(D)가 있으면 커밋 차단 + 안내 메시지
2. override: `ALLOW_RAW_DELETE=1 git commit ...` 환경변수로만 통과 (운영자 의도적 삭제용)
3. 설치: `git config core.hooksPath tools/hooks` 1회 — 훅 파일이 repo에 커밋되므로 클론해도 유지
4. CLAUDE.md git 규칙 절에 1줄 반영

- 공수: 셸 스크립트 ~15줄 (~15분)
- 시점: 바로 가능

## C. 페이지별 "마지막 갱신일" 표시 📅

**문제** — 팀원이 "이 페이지 최신 맞아?"를 물어봄. 신선도가 안 보임.

**계획**
1. `requirements.txt`에 `mkdocs-git-revision-date-localized-plugin` 추가
2. `mkdocs.yml` plugins에 등록 (`type: date`, `locale: ko`)
3. 주의: docs_dir가 심볼릭이라 git 이력 추적이 원본(documents/) 기준으로 잡히는지 로컬 확인 필요. CI에선 `actions/checkout`에 `fetch-depth: 0` 추가(이력 전체 필요)

- 공수: 설정 몇 줄 + 검증 (~20분)
- 시점: 아무 때나. 심볼릭 문제로 날짜가 이상하면 도입 보류(강행 안 함)

## D. Gate별 스냅샷 태그 🏷️

**문제** — "그 시점에 뭐가 확정이었나"를 나중에 되짚기 어려움.

**계획** — Gate 마일스톤마다 `git tag gate1 && git push origin gate1`. 오늘(7/7)이 Gate1이므로 **다음 커밋에 gate1 태그**. 이후 gate2(7/13)·final(7/29).

- 공수: 명령 1줄. /배포 절차에 "마일스톤이면 태그" 한 줄 추가 고려
- 시점: 다음 커밋 때 바로
